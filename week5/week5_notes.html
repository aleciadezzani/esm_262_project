Nesting Loops

Loops can be “nested” with one loop inside the other

Examples: - doing something to each row AND each column - doing something to each location AND each time step - doing something across categories AND through time

Syntax in R:

For each single step of the outer loop, the inner loop runs through all its iterations. So the loop above processes all columns for the first row, then moves to the next row. As before you don’t need to use i and j - these are just common variable names for counters but you can pick more meaningful ones

Complex Example of Nested Loops

Imagine we have running speeds of a species for different points in a transect. Observations are given to you in a matrix where rows are locations in the transect and columns are observations of speeds at each location for different individuals.

  1. Find number of times speeds were above the hunting threshold anywhere (easy)
  • just count all values in the matrix above threshold (e.g. sum(matrix >= threshold)) (could do without a loop)
  1. Find how often neighboring locations had speeds above the hunting threshold at the same time (harder)

  • need to loop through each location and observation and check if neighboring location also had a speed above the threshold at that time

  • loops make this easy because you can go back and forth in time with your counter!

#1 find number of times speed is above the hunting threshold

# create a sample data matrix
nobs = 4 #col
nlocation = 5 #rows

speedobs = matrix(runif(nobs*nlocation, min=0, max=100), nrow=nlocation)
colnames(speedobs) = c("ind1", "ind2", "ind3", "ind4")
rownames(speedobs) = c("0m", "20m", "40m", "60m", "80m")

# nested for loop: use o as counter for observation; l as counter for location

While loop

  • for = works with a counter to decide how many times to repeat

  • while = repeats until a condition is met

Goes through the loop until a condition is met

useful for when you don’t know how many times you need to repeat something but you want to keep going until a condition is met

often for simulations where you want to keep going until you reach a threshold or a certain outcome

Example of a While loop

Imagine trying to estimate how long it will take for a water quality threshold to be exceeded when there is a leak of pollution into a lake

You have an estimate of how much pollution is added each day, pol_in

You also know that the pollution is removed at a certain rate each day pol_out but removal efficacy depend on the air temperature that day:

\[ pol~{out} = pol~out,base~ ∗ (1 + 0.01∗air~temp~) \] You know air temperature is normally distributed mean 20 degrees Celsius and standard deviation of 5 degrees Celsius

you want to know how long it will take for the pollution to exceed a threshold

NOTE: I’m also going to add a check to stop if too many days pass! (we will talk about why)

# initialize parameters
pol_in = 10 # pollution added each day
pol_out_base = 5 # base pollution removed each day
threshold = 100 # pollution threshold

# initialize pollution level and day counter
pollution_level = 0
day = 0

maxday =365
# We need a while loop to simulate pollution levels over time
# we use && for AND because we want to check both conditions - that pollution level is below threshold and that we haven't exceeded max days

while ((pollution_level < threshold) && (day < maxday)) {
  # update pollution level
  air_temp = rnorm(mean=20, sd=5, n=1)
  pol_out = pol_out_base * (1 + 0.01*air_temp) # calculate pollution removed based on air temp
  pollution_level = pollution_level + pol_in - pol_out # update pollution level
  day = day + 1 # increment day counter
}
cat(sprintf("It takes %d days for pollution to exceed the threshold of %d, final level: %.1f", day, threshold, pollution_level))
It takes 25 days for pollution to exceed the threshold of 100, final level: 100.4
WarningWhy do we need a check for max days?

What if pollutant level never exceeds threshold? ==> Infinite Loop!

You can often use ‘ctrl C’ to break out of an infinite loop

It is better to have a check in your code to stop if you exceed a certain number of iterations

Functions (Makeup Lecture)

Modularity

Programs are made up of building blocks

A main program is a central piece of code that controls the flow and calls functions/modules/building blocks

Breaking tasks into functions is a central programming concept

Often modules are called ‘functions’ but also called: * ‘procedures’ * ‘subroutines’ * ‘methods’ * ‘subprograms’

You have already been using functions in R - e.g. mean(), lm(), plot() etc.

EXAMPLES of Reuse

  • an ecosystem model might re-use instructions for calculating how a living organism grows

  • e.g you might have a function that calculates growth from temperature and moisture and you can re-use this function for different organisms or different species with slightly different inputs

  • an accounting program might re-use instructions for computing net present value from interest rates, that you can re-use for different projects or different time periods

Modules often become ‘black boxes’ which hides detail that might make understanding the program overly complex

Most languages have lots of black boxes already written AND most allow you to write your own

Inputs and Parameters

All functions/modules have inputs/parameters and outputs

Inputs — what you give the function - sometimes separated into input data and parameters * input data = the “what” that is manipulated * parameters determine “how” the manipulation is done

  1. break your tasks into boxes with inputs, parameters and output
  2. WRITE DOWN the contract for each box/function
  • CONTRACT = if the users gives you A, the function with return B
  • specify A and B as precisely as you can (include units, time/space scale, parameters, options)

Building and Using a Function

Lets say we want to develop a function that computes power generated from hydro-reservoir. We know that power generation depends on water release rates and the potential energy (from the reservoir height).

Contract Input: Reservoir height (m) and flow rate (m3/sec) (these inputs could be a single value in time or a time series)

Output: Instantaneous power generation (W/s) (as a single value or value for each reservoir height and flow rate combination)

Parameters: KEff reservoir efficiency, ρ (density of water), g (acceleration due to gravity)

What the function will do compute power using the following equation - body of the function

P=ρ∗h∗r∗g*KEff

P is Power in watts, ρ is the density of water (~1000 kg/m3) h is height in meters, r is flow rate in cubic meters per second, g is acceleration due to gravity of 9.8 m/s2, KEff is a coefficient of efficiency ranging from 0 to 1.

Note

Function Name = function(inputs, parameters) { body of the function (manipulation of inputs) return(values to return) }

  • Body includes all calculations between { and }
  • anything inside the body is not “seen” by the workspace so you can re-use variables
  • return tells R what the output is
library(here)
# source the function from a file - make sure you are in the right working directory
source(here("week5/R/power_gen_orig.R"))

# alternatively
# function definition in-line
power_gen_orig = function(height, flow, rho=1000, g=9.8, Keff=0.8) {
result = rho * height * flow * g * Keff
return(result)
}

# note that height and flow don't have default values, so you must provide them when you use the function, but the other parameters have default values that you can overwrite if you want to

Using the function and scoping To use the function, we substitute “actual” values for each argument (parameter and inputs) (unless there are defaults). R assumes that arguments are in the order that they were specified in the function definition - unless you refer to them by name

Example:

Scoping

The scope of a variable in a program defines where it can be “seen” Variables defined inside a function cannot be “seen” outside of that function

There are advantages to this - the interior of the building block does not ‘interfere’ with other parts of the program

Some Conventions & Best Practices…

  • Always write your function in a text editor and then copy into R
  • By convention we name files with functions in them by the name of the function.R e.g. power_gen_orig.R
  • you can have R read a text file by source(“power_gen.R”) - make sure you are in the right working directory
  • keep organized by keeping all functions in a subdirectory called R
  • Include comments both in the body of the function and at the top - there is a particular style of adding comments to the top of the function that we will use later to generate automatic documentation - so useful to start to follow it, more or less, now - see power_gen_orig.R for the style
  • Eventually we will want our function to be part of a package (a library of many functions) to create a package you must use this convention (name.R) place all function in a directory called “R”
